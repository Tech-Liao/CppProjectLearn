# IM-server

V1-V5代表第1天至第5天的主要内容：基于socket+epoll实现一个即时聊天服务器的reactor网络模型。

它主要包括：tcpserver、connection、eventloop、buffer、codex类。

tcpserver主要负责作为总管家角色，它创建socket，实现非阻塞与端口复用功能。当有访客来访时通知connection进行连接处理。

connection进行非阻塞读取，将访客信息全部读取。

eventloop是监听者，监听epoll_fd，当epoll_fd有动静就会通知connection。其次，它封装了epoll相关基础接口。

buffer是作为数据接收发送的缓冲区，因此tcp是一个字节流，需要用一个容器把它装好。

codex是解决tcp的沾包与半包问题的。

### 一、 彻底搞懂“回调函数 (Callback)”

在传统的同步编程中，是我们主动去调用系统的函数，死死等它返回结果（比如阻塞的 `accept` 和 `read`）。 而在 Reactor 模型中，核心思想变成了著名的**好莱坞原则 (Hollywood Principle)："Don't call us, we'll call you."（不要打电话给我们，我们会打给你）。**

**生活化比喻：点外卖**

1.  **不用回调（阻塞模式）**：你去餐厅点餐，然后傻站在出餐口一直等。在此期间，你什么都干不了。
2.  **使用回调（Reactor 模式）**：你用手机点餐，并在订单上**留下了你的电话号码（这就是注册回调函数）**。然后你该打游戏打游戏，该看书看书（EventLoop 在循环）。等外卖做好了，**骑手拨打你的电话（这就是触发回调函数）**，你接到电话下楼拿外卖（执行你的业务逻辑）。

**映射到我们的代码中：**

-   `EventLoop` 就是那个无情的“外卖系统”，它只负责盯着谁的饭做好了（Epoll 监听）。
-   但 `EventLoop` 根本不知道饭做好了该给谁送！所以，我们在把文件描述符（fd）交给它的时候，必须附带一张**“纸条”（回调函数 `std::function`）**。

具体表现为两张极其关键的“纸条”：

1.  **TcpServer 递给 EventLoop 的纸条**：“这是 `listen_fd`。如果它响了，说明有新客人来了，请执行纸条上的逻辑（调用我的 `accept`，创建出 `Connection`）。”
2.  **Connection 递给 EventLoop 的纸条**：“这是我的 `client_fd`。如果它响了，说明客人发消息了，请执行纸条上的逻辑（调用我的 `Read` 函数去吸干数据）。”

如何从0实现

**第一步：我需要一个服务器来接客 (TcpServer)**

-   既然要接客，总得有个门面吧？于是写下 `socket()`, `bind()`, `listen()`。
-   *问题来了：* 客人来了我怎么接？传统的 `accept()` 会阻塞，导致我没法同时服务 10000 个人。
-   *推演结果：* 我需要一个能帮我同时盯着 10000 个人的工具。

**第二步：引入监控引擎 (EventLoop)**

-   C++ 在 Linux 下最高效的监控工具就是 Epoll。
-   于是封装一个 `EventLoop` 类，里面放一个 `epoll_fd`，写一个死循环 `epoll_wait` 一直等动静。
-   *问题来了：* 引擎发现动静后，怎么通知业务层？
-   *推演结果：* 引入回调机制！在 `EventLoop` 里加一个 `std::map<int, std::function>`。把 fd 存进去的同时，把想干的事也存进去。

**第三步：接客后的专项服务 (Connection)**

-   `TcpServer` 用回调执行了 `accept`，拿到了客人的 `client_fd`。
-   `TcpServer` 作为一个总管，不可能亲自去给客人端茶倒水（读写数据）。
-   *推演结果：* 必须 `new` 一个独立的 `Connection` 对象，把 `client_fd` 扔给它。`Connection` 诞生后的第一件事，就是去 `EventLoop` 那里登记自己的回调函数：“有数据了叫我”。

**第四步：应对残缺不全的数据 (Buffer & Codec)**

-   `Connection` 接到通知去 `read` 数据时发现，由于 TCP 是流，有时候读了一半，有时候读了两个包黏在一起。
-   *推演结果：* 必须有个容器先把数据兜住，于是写了 `Buffer`（里面包了个 `std::string`）。
-   然后必须制定规矩怎么切分这个容器里的数据，于是写了 `Codec`（TLV 协议，先读 8 字节头部算长度，再切包体）。
